# 事件

EventPoller 核心机制
> 绑定 -> 写入 -> 唤醒 -> 消费 -> 异常恢复

_event_map 用于存储文件描述符以及对应的回调  
_list_task 用于存储通过 async/async_first 提交的异步任务  



```text

绑定发生在构造其实
1. EventPoller::EventPoller() 调用 addEventPipe()
2. addEventPipe() 先把 _pipe.readFD()、_pipe.write() 设为非阻塞
3. 然后把 readFD() 注册到事件循环  addEvent(_pipe.readFD(), EventPOller::Event_Read, [this](int event){onPipeEvent();})
```

异步写入(写端) async_l
```text
1. 任务写入 _list_task  (加锁)
2. _pipe.write("", 1);写1字节

```

_pipe.read(buf, sizeof(buf)) 唤醒 runLoop 触发 回调函数，消费

> 读到数据就继续读，直到EAGIN，把管道彻底 drain 掉，避免频繁重复触发

为什么要“读空为止”
因为写端可能在短时间内写很多次(很多线程并发 async_l) ，如果只读一次：
- 管道里还残留字节
- poller 会马上再次被同一个fd 触发
- 造成无意义的 wakeup 风暴

消费
> 任务从管道事件转为执行

onPipeEvent(_ 读空以后并不在锁内逐个执行，而是:
1. _list_task 与本地 _list_swap 交换(短锁)
2. 锁外遍历执行 ——list_swap

好处：
- 缩短临界区
- 允许其他线程继续往 _list_task 投任务
- 批量执行，降低锁竞争 

异常恢复
当管道异常(失效)，执行:
1. delEvent(_pipe.readFD())
2. _pipe.reOpen(0
3. addEventPipe() 重新绑定

```cpp

inline void EventPoller::onPipeEvent(bool flush){
  char buf[1024];
  int err = 0;
  if(!flush){
      for(;;){
        if((err = _pipe.read(buf, sizeof(buf))) > 0){
          continue;
        }
        if(err == 0 || get_uv_error(true) != UV_EAGAIN){
          ErrorL <<"Invalid pipe fd of event poller, reopen it";
          delEvent(_pipe.readFD);
          _pipe.reOPen();
          addEventPipe();
      }
      break;
    }
  }

  decltype(_list_task) __list_swap;
  {
    lock_guard<mutex> lck(_mtx_task);
    __list_swap.swap(_list_task);
  }

    _list_swap.foreach([&] (const Task::Ptr &task){
      try{
        (*task)();
      }catch(ExitException &){
        _exit_flag = true;
      }catch(std::exception &e){
        ErrorL <<" Exception occurred when do asyc task:"<<ex.what();
      }
  });
}
```
