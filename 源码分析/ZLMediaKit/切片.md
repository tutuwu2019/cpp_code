# 切片

> 第一次听到切片这个概念是，一些博主的直播视频被剪切成若干个短视频放在短视频平台发布。还有就是很多新闻上出现有政论性的东西，说发布者刻意切片有争议性的主观臆断的视频，用来博取噱头等等。

是的，在音视频开发上切片，也是讲视频源文件切成若干个小的片段。小的片段，在播放效果上好于直播，这是我在ZLM 实际测试上的感受，比如音频和视频播着播着不同步，声音快视频一小会，然后刷新又同步上了。

然后采用切片会涉及到 cookie ，这是因为 cookie 能够保证一段时间http 的通信是可靠的。(安全)


播放器每次请求 .m3u8、.ts 切片、或建立 FLV 长连接时，都会触发hook，如果每个请求都去调业务服务器鉴权，就会有两个问题：
1. 性能问题：HLS每个几秒一个请求，hook 调用量会爆炸
2. 语义问题：同一个“播放会话”被拆成了无数次鉴权


```text
播放器                ZLMediaKit           业务服务器
  │                      │                    │
  │── GET /live/a.m3u8 →│                    │
  │                      │── on_play Hook ──→│  第1次鉴权
  │                      │←── {code:0} ──────│
  │── GET /live/a0.ts ──→│                    │
  │                      │── on_play Hook ──→│  第2次鉴权（重复！）
  │                      │←── {code:0} ──────│
  │── GET /live/a1.ts ──→│                    │
  │                      │── on_play Hook ──→│  第3次鉴权（重复！）


```

cookie 机制就能解决这两个问题，一次鉴权，cookie 保持会话，后续请求免鉴权。

> cookie 在性能上(减少hook 的调用) 和安全(会话级鉴权)之间做了一个精妙的平衡。




示例：交互过程


第一次请求（触发hook 鉴权）

```text
播放器                    ZLMediaKit                业务服务器
  │                          │                          │
  │─ GET /live/stream.m3u8 →│                          │
  │  (无 Cookie)             │                          │
  │                          │──── on_http_access ────→│
  │                          │     {ip, path, params}   │
  │                          │                          │ 验证 token
  │                          │←── {code:0,             │
  │                          │     cookie: "jwt=xxx",   │
  │                          │     second: 600} ────────│
  │                          │                          │
  │  HTTP 200                │
  │← Set-Cookie: jwt=xxx ───│
  │  (返回 m3u8 内容)        │

```


后续请求(cookie 命令，直接放行)

```text
播放器                    ZLMediaKit                业务服务器
  │                          │                          │
  │─ GET /live/a0.ts ───────→│                          │
  │  Cookie: jwt=xxx         │                          │
  │                          │  本地查找 Cookie 缓存     │
  │                          │  命中 → 直接放行          │
  │                          │  ✗ 不触发 Hook            │
  │← HTTP 200 ───────────────│                          │


```


cookie 过期，重新鉴权

```text
播放器                    ZLMediaKit                业务服务器
  │                          │                          │
  │─ GET /live/a100.ts ─────→│                          │
  │  Cookie: jwt=xxx         │                          │
  │                          │  Cookie 已过期            │
  │                          │──── on_http_access ────→│  重新鉴权
  │                          │←── {code:0, ...} ────────│

```

hook 返回字段解析

```text
{
  "code": 0,
  "msg": "success",

  "cookie": "userId=123&role=vip&streamId=live/test",
  // ↑ 写入 ZLM Cookie 缓存的 key，同时作为 Set-Cookie 返回给客户端
  // 可以是任意字符串，ZLM 以此为 key 做本地缓存

  "second": 600,
  // ↑ Cookie 有效期（秒）
  // 600 = 10分钟内不再触发 Hook
  // 0   = 每次请求都触发 Hook（禁用缓存）
  // -1  = 永久有效（慎用）

  "path": "/live/"
  // ↑ Cookie 作用路径
  // 只有访问 /live/ 下的资源才使用此 Cookie
  // 精细化控制：防止一个 Cookie 放行所有路径
}
```

### `path` 字段的语义陷阱
```
Cookie path = "/live/"

✅ /live/stream.m3u8     → 命中，直接放行
✅ /live/stream/a0.ts    → 命中，直接放行
❌ /record/2024/a.mp4    → 不命中，重新触发 Hook
❌ /hls/other.m3u8       → 不命中，重新触发 Hook
```

这意味着你可以**用一个 Cookie 覆盖一个频道的所有资源**，而不是整个服务器。

## Cookie 在不同协议中的行为差异

### HTTP 类协议（HLS / HTTP-FLV / MP4）

标准 HTTP Cookie 语义，浏览器自动携带：
```
GET /live/stream.m3u8
Cookie: zlmediakit_cookie=xxx
```

### RTSP / RTMP

这两个协议本身**没有 Cookie 概念**，ZLMediaKit 的处理方式：
```
rtmp://server/live/stream?token=abc123&sign=xyz
                          └─────────────────────┘
                            params 字段传入 Hook
                            业务侧用 token 鉴权即可

```


webRTC 通过 http 信令阶段携带 cookie 或 token，鉴权在信令层完成

```json

// ZLMediaKit → 业务服务器
{
  "mediaServerId": "zlm_node_01",
  "app":    "live",
  "stream": "camera_001",
  "schema": "http",
  "ip":     "10.0.0.55",

  "params": "token=eyJhbGci...&t=1704067200",
  // ↑ URL 问号后的参数，用于携带鉴权凭证

  "path":   "/live/camera_001/hls.m3u8",
  // ↑ 访问的具体路径

  "is_dir": false
  // ↑ 访问的是目录还是文件
  //   true  → 访问目录（如 /record/2024/）
  //   false → 访问具体文件
}

```
