
# 基础组件

## 线程

Thread 模块 是 ZLToolKit 统一任务执行抽象层

核心目标：
- 给上层 EventPollerPool、WorkThreadPool、网络会话 提供统一的  async/sync 调度接口
- 把线程管理、任务队列、负载感知、优先级、亲和性 封装成可复用的基础设施
- 让 IO 线程模型(EventPoller) 和 通用工作线程模型(ThreadPool) 共享同一套选择器(TaskExecutorGetterImp)

ZLM 线程模型本质是：
Reactor + 线程池 + 任务队列

当然具体的，ZLM中核心线程主要有这几类：
1. PollThread 网络事件线程
2. WorkThread 业务线程
3. TimerThread 定时器线程
4. 少量普通线程 (日志/后台任务/启动线程)

具体模块拆分

1. TaskExecutor/TaskExecutorInterface：统一执行契约
   抽象接口:
     - async(task, may_sync)
     - async_first(task, may_sync) 高优先级
     - sync/sync_first (通过信号量同步等待)
  
   TaskCancelableImp:
     - 任务可取消 cancel()
     - 通过 weak_ptr 防止执行已取消任务
   may_sync 语义：
     - 如果当前就在目标执行器线程，允许直接执行，避免无意义线程切换

> 设计思想：用一层统一屏蔽“到底是 poller 线程还是普通线程池"，上层代码只关系"投递任务"


2. ThreadLoadCounter： 轻量化负载感知

  - 用"运行时间/休眠时间" 窗口估算线程负载(0~100)
  - startSleep()/ sleepWakeUp() 在任务循环循环中埋点
  - 按样本数 + 时间窗口双重裁剪，避免历史污染

> 设计思想：不是追求绝对CPU 监控精度，而是用于调度决策的近似指标，成本低、足够实用


3. TaskExecutorGetterImp ： 执行器选择策略核心

它维护 _threads (TaskExecutor::Ptr 列表)，提供三类选路：
  - getExecutor(): 按负载选择最轻执行器(带轮询起点，避免总挑第一个)
  - getExecutor(cb): 异步"延迟优先"选择(向每个执行器投一个探测任务，谁先跑到回调谁优先)
  - getExecutorDelay():采样每个执行器排队延迟

并且负责 addPoller():
  - 创建 EventPoller
  - 启动 loop
  - 异步设置线程名/CPU 亲和性

> 设计思想: 把"线程池管理"升级为"可观测+ 可策略选择" 的执行器路由层


4. TaskQueue<T>: 简洁阻塞队列

  - push_task / push_task_frist
  - get_task （信号量阻塞等待）
  - push_exit(n) 投放 n 个 “推出信号" (空任务)


> 设计思想: 极简但够用；不高复杂 lock-free，优先稳定和可读性



5. ThreadPool: 通用线程池实现
  - 继承 TaskExecutor，实现 async/async_first
  - run() 主循环:
    - startSleep() -> get_task() -> sleepWakeUp()
    - 执行任务并捕获异常
   
  - 析构时，shutdown + join_all
  - 执行线程优先级、线程名、CPU亲和性

> 设计思想: 把"线程池"实现为 TaskExecutor 的一种具体后端，而不是另一套接口体系

6. WorkThreadPool： 工作型 Poller池(而非普通ThreadPool)

  - 继承 TaskExecutorGetterImp
  - 内部实际放的是 EventPoller（通过 addPoller)
  - 默认低优先级 work poller
  - getPoller() 实际走 getExecutor() 策略

> 设计思想: ZLToolKit 里"工作线程"也秦香统一为 poller 线程模型，减少两套模型割裂

7. semaphore / threa_group： 并发基础原语

  - semaphore.h:
    - 有系统 sem 用 sem，没有则 condition_variable_any 回退
    - 提供超时等待
  - threadgroup.h
    - 管理线程集合
    - 防止线程自我 join (join_all() 安全保护)
   
> 设计思想: 跨平台 + 最小依赖，避免把并发控制分散到各个模块里重复实现


总结： 

EventPoller 直接继承 Taskexecutor 
EvenPollerPool 继承 TaskExecutorGetterImp 

这意味着：
  - IO 线程和工作线程在抽象层是同构的(都能 async/sync)
  - 调度器可复用(负载选路、延迟选路)
  - 上层业务代码不需要知道"我现在是在epoll 线程还是在普通线程池"

> 这是ZLM 能保持高并发同时代码可维护的关键架构点


优点：
- 统一抽象强：任务投递接口一致
- 调度只能：支持按负载和延迟选执行器
- 工程实用：线程名、优先级、亲和性都内置
- 线程切换可控：may_sync 避免不必要上下文切换


代价/风险点：
- 负载统计是近似值，不是OS 级精确CPU计量
- TaskQueue 是mutex + semaphore,极端场景下吞吐不如 lock-free 队列
- semaphore::wait(timeout) 在HAVE_SE 分支里有重复 timespec 代码块(可读性问题，后续可清理)


## poller

在ZLToolKit/ZLM里，EventPoller 不是 epoll 的薄封装，而是一个线程内运行时(thread runtime):
  - 管IO事件(addEvent/delEvent/modifyEvent)
  - 管跨线程任务投递(async/async_first)
  - 管定时任务(doDelayTask)
  - 管线程本地上下文(AnyStoreage)
  - 管该线程共享资源(getShareBuffer)


> 它相当于： epoll/kqueue/select + task loop + timer wheel(map) + thread-local context


1. One Loop One Thread （单Poller 单线程)

- _loop_thread
- _lost_task (跨线程投递过来的任务)
- _delay_task_map
- _event_map

> 这些结构默认都在 poller 所在线程消费。意义：把并发问题转成"串行化事件处理"，大量业务对象可以无锁/少锁。这也是为什么有大量 API 会有 map_sync、isCurrentThread()，如果已经在本线程，直接执行，避免线程跳转和锁


2. IO 与任务调度一致，而不是两套系统

很多库把网络事件循环和业务线程池拆成两套;ZLM 选择同一道 TaskExecutor 语义：
- EventPoller 继承 TaskExecutor
- EventPollerPool 继承 TaskExecutorGetterImp

这意味着上层代码不需要关心 这是 IO 线程还是 worker 线程，只要 async(...) 即可

统一调度语义带来两个收益：
- 代码一致性高
- 任务可以自然会到所属poller 线程，保持线程亲和性


3. AnyStoreage = Poller 线程级上下文容器

- 每个 poller 线程都可以挂自己的对象(缓存、统计器、状态)
- 避免全局单例污染
- 避免每次调用层层穿参

4. 共享读缓存减少分配抖动

_shared_buffer[2] + getSharedBUffer(bool is_udp) 表明：

- 每个poller 线程维护 socket 读缓冲共享池(TCP/UDP分开)
- 在高并发连接下减少频繁 malloc/free
- 减少 cache miss 和内存碎片
