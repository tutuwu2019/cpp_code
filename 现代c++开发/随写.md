# 智能指针

```cpp
// srt/SrtTransportImp.cpp

void SrtTransportImp::emitOnPublish() {
    // ① 从 shared_from_this() 取强引用再转为弱引用
    std::weak_ptr<SrtTransportImp> weak_self =
        static_pointer_cast<SrtTransportImp>(shared_from_this());

    Broadcast::PublishAuthInvoker invoker = [weak_self](const std::string &err, const ProtocolOption &option) {
        auto strong_self = weak_self.lock(); // ② 第一次 lock（鉴权回调线程）
        if (!strong_self) { return; }

        strong_self->getPoller()->async([weak_self, err, option]() {
            auto strong_self = weak_self.lock(); // ③ 第二次 lock（IO 线程）
            if (!strong_self) { return; }
            // 安全操作 muxer
            strong_self->_muxer = std::make_shared<MultiMediaSourceMuxer>(...);
        });
    };
}

```

为什么 要把 shared_ptr 转成 weak_ptr ？ 这不是多此一举吗？
```cpp
void SrtTransportImp::emitOnPublish() {
    // 错误写法：直接捕获 shared_ptr
    auto self = static_pointer_cast<SrtTransportImp>(shared_from_this());

    Broadcast::PublishAuthInvoker invoker = [self](...) {
        // invoker 持有 self 的强引用
        // ↑ 哪怕客户端断连、SrtTransportImp 应该销毁，
        //   只要 invoker 还活着，引用计数就不会归零，对象永远不会析构！
        self->getPoller()->async([self]() {
            // ...
        });
    };
    // invoker 被传给外部，可能存活很长时间...
}
```

后果：
```text
客户端断连
  └─ SrtTransport 应该析构
       └─ 但 invoker 还被外部持有 (比如你的鉴权服务器还没回调)
            └─ invoker 里的 shared_ptr self 阻止了析构
                 └─ 内存泄漏 + 状态残留（muxer、reader 不会被清理）
```

正确处理:
```tet
客户端断连
  └─ SrtTransport 析构 ✓（引用计数降为 0，因为 weak_ptr 不增加计数）
       └─ 稍后鉴权回调触发 invoker
            └─ weak_self.lock() 返回 nullptr
                 └─ 立刻 return，什么都不做 ✓
```

 总结：内部嵌套获取当前资源，一定要确保当前的 资源是有效的。简单来说就是 获取之前要询问
