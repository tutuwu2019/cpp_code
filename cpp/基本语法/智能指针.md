# 智能指针
## 使用与分析
```text
讲讲智能指针                                                                                                             
                       
● 为什么需要智能指针                                                                                                       
                                                                                                                           
  裸指针的问题：                                                                                                           
                                                                                                                           
  void foo() {                                                                                                             
      Buffer* p = new Buffer(10);                                                                                          
      // ... 中间如果抛异常，或者 return 提前退出
      delete p;  // 可能永远执行不到 → 内存泄漏
  }

  手动 new/delete 需要人来保证配对，很容易出错。

  智能指针的本质：把指针包在一个对象里，利用析构函数自动释放资源（RAII）。

  ---
  unique_ptr —— 独占所有权

  ▎ 一块内存只能有一个主人。

  #include <memory>

  std::unique_ptr<Buffer> p = std::make_unique<Buffer>(10);
  // 离开作用域自动 delete，不需要手动管理

  不能拷贝，只能移动：

  auto p1 = std::make_unique<Buffer>(10);
  auto p2 = p1;              // 错：不能拷贝，所有权唯一
  auto p2 = std::move(p1);   // OK：转移所有权，p1 变成 nullptr

  移动前：p1 ──→ [Buffer]    p2 = nullptr
  移动后：p1 = nullptr       p2 ──→ [Buffer]

  常用操作：

  p.get();        // 获取裸指针（不转移所有权）
  p.release();    // 释放所有权，返回裸指针，p 变 nullptr
  p.reset();      // 销毁对象，p 变 nullptr
  p.reset(ptr);   // 销毁旧对象，接管 ptr

  ---
  shared_ptr —— 共享所有权

  ▎ 多个指针共同拥有一块内存，最后一个离开时释放。

  靠引用计数实现：

  auto p1 = std::make_shared<Buffer>(10);  // 引用计数 = 1
  {
      auto p2 = p1;  // 拷贝，引用计数 = 2
      auto p3 = p1;  // 拷贝，引用计数 = 3
  }   // p2, p3 析构，引用计数 = 1
  // p1 析构，引用计数 = 0 → 释放内存

  p1 ──┐
  p2 ──┼──→ [Buffer] + [count: 3]
  p3 ──┘

  常用操作：

  p.use_count();   // 当前引用计数
  p.unique();      // 是否只有一个引用

  ---
  shared_ptr 的陷阱：循环引用

  struct Node {
      std::shared_ptr<Node> next;
  };

  auto a = std::make_shared<Node>();
  auto b = std::make_shared<Node>();
  a->next = b;  // b 引用计数 = 2
  b->next = a;  // a 引用计数 = 2

  // a, b 离开作用域，引用计数各减到 1
  // 互相持有，永远不会到 0 → 内存泄漏

  a ──→ [NodeA] ──→ [NodeB] ──→ [NodeA] ...（死循环）

  ---
  weak_ptr —— 打破循环引用

  ▎ 观察 shared_ptr 管理的对象，但不增加引用计数，不参与所有权。

  struct Node {
      std::shared_ptr<Node> next;
      std::weak_ptr<Node> prev;   // 用 weak_ptr 打破循环
  };

  使用时需要先 lock() 升级成 shared_ptr：

  std::weak_ptr<Buffer> wp = p1;  // 不增加引用计数

  if (auto sp = wp.lock()) {  // 尝试升级
      // 对象还活着，sp 是有效的 shared_ptr
  } else {
      // 对象已经被销毁了
  }

  weak_ptr 不能直接访问对象，必须先 lock()，这也防止了悬空指针。

  ---
  三者对比

  ┌──────────┬────────────┬──────────────────┬──────────────────────────┐
  │          │ unique_ptr │    shared_ptr    │         weak_ptr         │
  ├──────────┼────────────┼──────────────────┼──────────────────────────┤
  │ 所有权   │ 独占       │ 共享             │ 无                       │
  ├──────────┼────────────┼──────────────────┼──────────────────────────┤
  │ 引用计数 │ 无         │ 有               │ 不增加计数               │
  ├──────────┼────────────┼──────────────────┼──────────────────────────┤
  │ 能拷贝   │ 否         │ 是               │ 是                       │
  ├──────────┼────────────┼──────────────────┼──────────────────────────┤
  │ 能移动   │ 是         │ 是               │ 是                       │
  ├──────────┼────────────┼──────────────────┼──────────────────────────┤
  │ 主要用途 │ 单一所有者 │ 多个所有者       │ 打破循环引用/缓存/观察者 │
  ├──────────┼────────────┼──────────────────┼──────────────────────────┤
  │ 性能开销 │ 几乎为零   │ 引用计数原子操作 │ 极小                     │
  └──────────┴────────────┴──────────────────┴──────────────────────────┘


  ---
  选择原则

  默认用 unique_ptr
      ↓ 需要共享所有权？
  shared_ptr
      ↓ 出现循环引用 或 只需要观察不需要所有权？
  weak_ptr

  ▎ 90% 的场景 unique_ptr 够用，过度使用 shared_ptr 会带来不必要的引用计数开销。

  ---
  make_unique / make_shared 而不是 new

  // 不推荐
  std::shared_ptr<Buffer> p(new Buffer(10));

  // 推荐：make_shared 把对象和控制块分配在一起，减少一次内存分配
  auto p = std::make_shared<Buffer>(10);

  make_unique / make_shared 还能避免裸 new 带来的异常安全问题，始终优先使用。

❯ 为什么用上了智能指针还是会出现double free 的问题？                                                                       
                                                     
● 根本原因：用智能指针包裹了同一个裸指针                                                                                   
                                                                                                                           
  智能指针的析构函数会 delete 它持有的裸指针。如果两个智能指针不知情地指向同一块内存，析构时就各 delete 一次。             
                                                                                                                           
  ---                                                                                                                      
  场景1：用同一个裸指针构造两个 shared_ptr                                                                                 

  Buffer* raw = new Buffer(10);

  std::shared_ptr<Buffer> p1(raw);  // 引用计数 = 1，控制块A
  std::shared_ptr<Buffer> p2(raw);  // 引用计数 = 1，控制块B（新建的！）

  // p1 析构 → 计数A归0 → delete raw
  // p2 析构 → 计数B归0 → delete raw  ← double free

  两个 shared_ptr 各自建了独立的控制块，互相不知道对方的存在：

  p1 ──→ [控制块A: count=1] ──→ [raw]
  p2 ──→ [控制块B: count=1] ──┘   ↑ 同一块内存

  修复：用 make_shared，或者拷贝 shared_ptr 而不是重新包裹裸指针：

  auto p1 = std::make_shared<Buffer>(10);
  auto p2 = p1;   // 共享同一个控制块，引用计数 = 2

  ---
  场景2：unique_ptr 和裸指针混用

  Buffer* raw = new Buffer(10);
  std::unique_ptr<Buffer> p(raw);

  delete raw;  // 手动 delete
  // p 析构时再次 delete → double free

  或者反过来：

  auto p = std::make_unique<Buffer>(10);
  Buffer* raw = p.get();   // 拿到裸指针
  delete raw;              // 手动 delete
  // p 析构时再次 delete → double free

  ---
  场景3：shared_ptr 和 this 指针的陷阱

  class Node {
  public:
      void addToList(std::vector<std::shared_ptr<Node>>& list) {
          list.push_back(std::shared_ptr<Node>(this)); // 用 this 构造新的 shared_ptr
      }
  };

  auto p = std::make_shared<Node>();  // 控制块A
  p->addToList(list);                 // 又建了控制块B，包的是同一个 this

  // p 析构 → 控制块A归0 → delete
  // list 里的 shared_ptr 析构 → 控制块B归0 → double free

  修复：继承 enable_shared_from_this：

  class Node : public std::enable_shared_from_this<Node> {
  public:
      void addToList(std::vector<std::shared_ptr<Node>>& list) {
          list.push_back(shared_from_this()); // 共享同一个控制块
      }
  };

  ---
  总结：double free 的根本模式

  ▎ 同一块内存，被两个独立的所有者各自认为归自己管。

  ┌───────────────────────────────┬────────────────┐
  │             场景              │      原因      │
  ├───────────────────────────────┼────────────────┤
  │ 同一裸指针构造两个 shared_ptr │ 两个独立控制块 │
  ├───────────────────────────────┼────────────────┤
  │ 智能指针 + 手动 delete 混用   │ 所有权不清晰   │
  ├───────────────────────────────┼────────────────┤
  │ 用 this 构造 shared_ptr       │ 控制块重复创建 │
  └───────────────────────────────┴────────────────┘

  根治方法只有一个：资源从创建起就交给智能指针，永远不把裸指针当所有者使用，get() 拿到的裸指针只用来"借用"，不 delete。

❯ 有的时候会因为  set 或者 reset 或者其他func  shared_ptr 而出现 double free                                               
                                                                                               
● 对，核心原因还是同一块内存被多个独立控制块管理。                                                                         
                                                                                                                           
  ---                                                                                                                      
  reset() 引发 double free                                                                   
                                                                                                                           
  Buffer* raw = new Buffer(10);                                                                                            
  auto p1 = std::shared_ptr<Buffer>(raw);
  auto p2 = std::shared_ptr<Buffer>(raw); // 独立控制块！

  p1.reset(); // 控制块A归0 → delete raw
  p2.reset(); // 控制块B归0 → delete raw ← double free

  reset() 本身没问题，问题是 p1 和 p2 从一开始就不共享控制块，reset 只是触发了早已埋下的炸弹。

  ---
  reset(ptr) 引发 double free

  auto p1 = std::make_shared<Buffer>(10);
  auto p2 = p1;  // 共享，引用计数 = 2

  Buffer* raw = p1.get();
  p1.reset(raw); // ← 危险！p1 放弃旧对象，然后用 raw 重新构造新控制块
                 // 但 p2 还持有旧控制块指向同一块内存

  // p2 析构 → 旧控制块归0 → delete raw
  // p1 析构 → 新控制块归0 → delete raw ← double free

  ---
  本质

  reset(ptr) 和 shared_ptr(ptr) 都会无条件新建控制块，不管 ptr 是否已经被其他智能指针管理。

  ▎ 只要用裸指针去构造或 reset 一个 shared_ptr，就有 double free 的风险。

  规则：

  make_shared     → 安全，控制块和对象一起创建
  shared_ptr(ptr) → 危险，除非 ptr 是刚 new 出来的，没有被任何智能指针管理
  reset(ptr)      → 同上
  shared_from_this → 需要继承 enable_shared_from_this 才能安全拿到 shared_ptr

```


## 源码分析


### 为什么要在shared_ptr 中引入 weak_ptr ?

这要从 根本原因：循环引用说起。

> 循环引用，当两个对象互相持有，资源最后无法释放。

```cpp
struct Node{
  std::shared_ptr<Node> next;
};

auto a = std::make_shared_ptr<Node>();
auto b = std::make_shared_ptr<Node>();

a->next = b;
b->next = a;

// 函数结束时，局部变量 a,b 析构
// a 的引用计数：2->1
// b 的引用计数：2->1
//两者计数都没有归零，谁也不发生析构，发生内存泄漏

```

发生循环引用的场景都有哪些？  特点，一种很明显的映射关系
1. 双向链表(prev/next)
2. 树的双向结构(parent/children)
3. 观察者模式(subject 持有 observers,observer 持有 subject)
4. 对象图(图里有环是常态)
5. GUI父子控件(parent持有子空间，子空间要回调 parent)
6. 缓存(cache 持有对象，对象持有指回的指针)  

简单来说，weak_ptr 就是为了解决 shared_ptr 中出现的循环引用的应运而生的。换言之，weak_ptr 单独使用毫无意义。具体来说，weak_ptr 就是这所有权和引用这种映射关系给解绑(解耦)。
从理解角度，weak_ptr 没有引用计数，但是从源码角度角度触发，shared_ptr 和 weak_ptr 是通过 控制块控制对象，具体来说，指针对象可以管理对象的完整生命周期，包括释放(通过strong_count 实现)，而控制块则是通过 weak_count 来维护(映射到weak_ptr)，具体来说，当weak_ptr 负责维护对象以及控制块。控制块。
完整的生命周期，当初始化一个 shared_ptr  strong_count 和 weak_count 都会+1，当拷贝或者复制 shared_ptr对象，那么 strong_weak 会自然的+1，weak_ptr 不会变化，当使用者逐渐减少 strong_count 也会随之减少。当strong_weak 和 weak_count 都减少为1 时，会先释放对象空间，再释放控制块，整个内存资源释放。但是为了解决引用计数的问题，当 shared_ptr 中的对象需要使用 weak_ptr 来维护，这里会有怎么样的操作呢？先是 weak_ptr 会 lock ，里面的具体实现包括：先是检查控制块是否存在，然后尝试 对 strong_count 检查，是否对象资源存在，如果存在，尝试获取所有权(转成shared_ptr，当然strong 引用计数也会+1)，后面的流程和上面的一样。

### 一些细节

#### 原子性的检查当前值是否等于期待值(expected)
```cpp
bool compare_exchange_strong(T& expected, T desired, ...);
bool compare_exchange_weak(T& expected, T desired, ...);

```
如果与期待值一直，那么就讲值替换为 desired ，并返回为true 否则把expected 更新为当前值并返回false

### 一些疑问以及回答

```text
long cur = strong_count.load(std::memory_order_relaxed);  // ①
while (cur != 0) {                                         // ②
    if (strong_count.compare_exchange_weak(cur, cur + 1,
            std::memory_order_acq_rel,                     // ③
            std::memory_order_relaxed))                    // ④
        return true;
    // CAS 失败时 cur 被自动更新为最新值，循环继续        // ⑤
}
return false;                                              // ⑥

① 用 relaxed 读初始值，因为后面 CAS 会做更强的同步，这里不需要 barrier

② 核心不变量：cur == 0 意味着对象已销毁，提升不可能成功，直接退出

③ CAS 成功时用 acq_rel：acquire 保证能看到之前所有对该对象的写入，release 保证新持有者可见

④ CAS 失败时只需 relaxed，失败代表有竞争，用 cur 的最新值重试，不需要 barrier

⑤ compare_exchange_weak 允许伪失败（spurious failure），失败时会把 cur 更新为内存中的实际值，下次循环用新值重试，比 strong 版本在有竞争时吞吐更高

⑥ 退出循环唯一条件是 cur == 0，对象已死，返回 false
```

问题二：acq_rel 如何保证可见性

```text
线程A（构造 SharedPtr）            线程B（WeakPtr::lock()）
─────────────────────────────     ──────────────────────────────
obj->data  = 42;        ─┐
obj->ready = true;       │  这些写操作被 release 封口
strong_count.store(1,    │
  memory_order_release) ─┘

                                  CAS 成功，memory_order_acquire ← 拆封
                                  // 此时保证能看到：
                                  // obj->data  == 42   ✓
                                  // obj->ready == true ✓
                                  use(*obj);

```
如果没有 acquire/release，编译器和CPU可能会把 obj->data = 42 地重写操作排到 strong_count 赋值之后，线程B看到strong_count > 0 时，obj->data 可能还是旧值。
acq_rel = CAS 成功时，同时做了 acquire+ release，因为提升方既要看到之前写的(acquire)，也要把自己的状态发布出去(release)
