# 基础语法


## 关于默认的6个成员函数以及移动语义

移动语义的本质是获取所有权。


默认构造、默认拷贝构造、默认拷贝赋值、默认移动构造、默认移动赋值、默认虚构。

由于，默认的拷贝构造、默认拷贝赋值、默认移动构造、默认移动赋值只是简单的绑定指针，而没有把原始对象置nullptr ，导致两个对象指向同一个内存，会造成 double free

```cpp
#include <iostream>

class Buffer{
    char* data_;
    size_t size_; 
public:
    Buffer(size_t n): data_(new char[n]), size_(n){
        std::cout << "Buffer constructor, size: " << size_ << std::endl;
    }
    // 拷贝构造
    Buffer(const Buffer& other): data_(new char[other.size_]), size_(other.size_){
        std::cout << "Buffer copy constructor, size: " << size_ << std::endl;
        std::copy(other.data_, other.data_ + size_, data_);
    }
    // 拷贝赋值运算符
    Buffer& operator=(const Buffer& other){
        std::cout << "Buffer copy assignment operator, size: " << other.size_ << std::endl;
        if(this != &other){
            delete[] data_; // 释放当前资源
            size_ = other.size_;
            data_ = new char[size_]; // 分配新资源
            std::copy(other.data_, other.data_ + size_, data_); // 复制数据
        }
        return *this;
    }
    // 必须实现移动构造函数和移动赋值运算符，否则会调用默认的拷贝构造函数和拷贝赋值运算符，导致 double free 错误
    Buffer(Buffer&& other) noexcept : data_(other.data_), size_(other.size_) {
        std::cout << "Buffer move constructor, size: " << size_ << std::endl;
        other.data_ = nullptr; // 转移资源所有权
        other.size_ = 0;
    }

    // 移动赋值运算符
    Buffer& operator=(Buffer&& other) noexcept {
        std::cout << "Buffer move assignment operator, size: " << other.size_ << std::endl;
        if (this != &other) {
            delete[] data_; // 释放当前资源
            data_ = other.data_; // 转移资源所有权
            size_ = other.size_;
            other.data_ = nullptr; // 转移资源所有权
            other.size_ = 0;
        }
        return *this;
    }
    // 必须手写析构函数
    ~Buffer(){
        std::cout << "Buffer destructor, size: " << size_ << std::endl;
        delete[] data_;
    }
};

int main(){
    Buffer buf1(10); // 调用构造函数
    Buffer buf2 = buf1; // 调用拷贝构造函数
    Buffer buf3(20);
    buf3 = buf1; // 调用拷贝赋值运算符

    Buffer buf4 = std::move(buf1); // 调用移动构造函数
    Buffer buf5(30);
    buf5 = std::move(buf2); // 调用移动赋值运算符

    return 0;
}
```

```text
所以移动语义的本质是获取原有对象的所有权？                                                                               
                                             
● 是的，移动语义的本质就是转移所有权。                                                                                     
                                                                                                                         
  更精确地说：                                                                                                             
                                                                                                                           
  ▎ 移动 = 从"即将消亡"的对象那里窃取资源，而不是复制资源。                                                                
                                                                                                                           
  ---
  所有权转移的三个动作

  以你的代码为例：

  Buffer(Buffer&& other) noexcept
      : data_(other.data_)   // 1. 我拿走你的指针
      , size_(other.size_)
  {
      other.data_ = nullptr; // 2. 你的指针清空（你不再拥有它）
      other.size_ = 0;       // 3. 你变成"空壳"
  }

  ┌─────────────────────────┬──────────────────────────┐
  │          动作           │           含义           │
  ├─────────────────────────┼──────────────────────────┤
  │ data_(other.data_)      │ 新对象获得资源所有权     │
  ├─────────────────────────┼──────────────────────────┤
  │ other.data_ = nullptr   │ 原对象放弃所有权         │
  ├─────────────────────────┼──────────────────────────┤
  │ 析构时 delete[] nullptr │ 空壳析构是安全的，无操作 │
  └─────────────────────────┴──────────────────────────┘

  ---
  与拷贝的对比

  拷贝：A ──→ [内存]        深拷贝后：A ──→ [内存A]
                                      B ──→ [内存B]（新分配）

  移动：A ──→ [内存]        移动后：  A ──→ nullptr（放弃）
                                      B ──→ [内存]（原来的）

  拷贝是"复印"，移动是"搬家"。

  ---
  为什么要有移动语义

  核心是避免不必要的深拷贝。对象在以下场景会"即将消亡"，资源完全可以直接转移：

  Buffer make_buffer() {
      Buffer tmp(1024);
      return tmp;           // tmp 即将销毁，直接把内存转给调用方
  }

  Buffer b = make_buffer(); // 移动构造，零次内存分配+复制

  如果没有移动语义，这里会 new 一块 1024 字节的新内存，把数据 memcpy 过去，再 delete 旧的——完全是浪费。

  ---
  一句话：移动语义让 C++ 能安全地说"这个对象我不要了，你拿去用"，省掉深拷贝的开销，同时通过置 nullptr 保证所有权唯一，避免
  double free。
```

```text
如何理解将亡值？常见的有左值和右值、左值引用和右值引用还有将亡值。我对这几个值不是很清楚能不能用最透彻的话讲清楚，可以适 
当的举例子。可以展开说说                                                                                                   
                                                                                                                           
● 从"能不能取地址"开始                                                                                                     
                                                                                                                           
  C++ 里每个表达式都有两个属性：类型 和 值类别。值类别回答的是：                                                           
  ▎ 这个表达式代表的东西，有没有稳定的内存地址？                                                                           
                                                                                                                           
  ---                                                                                                                      
  左值（lvalue）  

  有名字、有地址、能持续存在的对象。

  int x = 10;
  x;          // 左值：有名字，有地址，&x 合法
  buf1;       // 左值：Buffer 对象，有地址

  直觉：能放在赋值号左边的，基本都是左值（名字来源于此，但不完全准确）。

  ---
  纯右值（prvalue）

  临时计算结果，没有内存地址，表达式结束就消失。

  10;         // 纯右值：字面量，没有地址
  x + 1;      // 纯右值：计算结果，存在寄存器里，没有稳定地址
  Buffer(10); // 纯右值：匿名临时对象

  直觉：不能取地址，不能放左边。

  ---
  将亡值（xvalue，expiring value）

  这是最难理解的一个，先说结论：

  ▎ 将亡值 = 有地址，但你主动声明"我不再需要它了，可以抢走它的资源"的对象。

  它是左值和右值之间的桥梁：
  - 像左值：有内存地址，对象实体存在
  - 像右值：资源可以被转移，即将"消亡"

  Buffer buf1(10);
  std::move(buf1);   // ← 这就是将亡值

  std::move 没有移动任何东西，它只是做了一个强制类型转换，把
  buf1（左值）转成将亡值，告诉编译器："这个对象的资源你可以抢走"。

  ---
  三者关系总览

  所有表达式
  ├── 泛左值 glvalue（有地址）
  │   ├── 左值 lvalue     → 有名字，长期存在
  │   └── 将亡值 xvalue   → 有地址，但资源即将被转移
  └── 右值 rvalue（资源可被转移）
      ├── 将亡值 xvalue   → 同上（两边都有它）
      └── 纯右值 prvalue  → 无地址的临时值

  将亡值同时属于"泛左值"和"右值"，是两个集合的交集。

  ---
  引用的本质：绑定到不同值类别

  左值引用 T&

  只能绑定左值：

  int x = 10;
  int& ref = x;       // OK：x 是左值
  int& ref2 = 10;     // 错误：10 是纯右值
  int& ref3 = x + 1;  // 错误：x+1 是纯右值

  右值引用 T&&

  绑定右值（纯右值 或 将亡值）：

  int&& r1 = 10;              // OK：10 是纯右值
  int&& r2 = x + 1;           // OK：x+1 是纯右值
  int&& r3 = std::move(x);    // OK：std::move(x) 是将亡值

  int&& r4 = x;               // 错误：x 是左值

  const T& 万能引用（只读）

  历史遗留，能绑定一切：

  const int& r = 10;   // OK：const 左值引用可以绑定右值

  ---
  关键陷阱：右值引用变量本身是左值

  void foo(Buffer&& buf) {
      // buf 的类型是 Buffer&&（右值引用）
      // 但 buf 这个变量本身是左值！因为它有名字、有地址

      Buffer b = buf;            // ← 调用拷贝构造！不是移动！
      Buffer b2 = std::move(buf); // ← 才是移动构造
  }

  ▎ 有名字的右值引用变量，是左值。

  这是很多人困惑的地方。std::move 的作用就是把它重新转回将亡值。

  ---
  完整例子串联起来

  Buffer buf1(10);               // buf1：左值

  Buffer buf2 = buf1;            // buf1 是左值 → 拷贝构造

  Buffer buf3 = Buffer(20);      // Buffer(20) 是纯右值 → 移动构造（或直接构造）

  Buffer buf4 = std::move(buf1); // std::move(buf1) 把左值转成将亡值 → 移动构造
                                 // buf1 之后变成"空壳"，data_==nullptr

  ---
  一张图总结

                      有地址？
                     /        \
                   是            否
                   |             |
                泛左值         纯右值 (prvalue)
               /      \        字面量、临时计算结果
             左值     将亡值 (xvalue)
          有名字的    std::move(x)
          普通对象    函数返回的 T&&

          ←────────────────────────────→
          资源不可转移              资源可转移
          (拷贝)                    (移动)

  ---
  一句话记住每个概念

  ┌──────────────┬──────────────────────────────────────┐
  │     概念     │                一句话                │
  ├──────────────┼──────────────────────────────────────┤
  │ 左值         │ 有名字有地址，活得好好的             │
  ├──────────────┼──────────────────────────────────────┤
  │ 纯右值       │ 临时的计算结果，转瞬即逝             │
  ├──────────────┼──────────────────────────────────────┤
  │ 将亡值       │ 有地址，但主动宣布放弃所有权         │
  ├──────────────┼──────────────────────────────────────┤
  │ 左值引用 T&  │ 只绑左值，保证对象活着               │
  ├──────────────┼──────────────────────────────────────┤
  │ 右值引用 T&& │ 绑右值/将亡值，可以抢资源            │
  ├──────────────┼──────────────────────────────────────┤
  │ std::move    │ 不移动任何东西，只是把左值转成将亡值 │
  └──────────────┴──────────────────────────────────────┘
```

```text
如何理解这句话：引用的本质：绑定到不同值类别                                                                             
                                                                                                                           
● 先理解"绑定"是什么意思                                                                                                   
                                                                                                                           
  "绑定"就是给一块内存起别名。                                                                                             
   
  int x = 10;                                                                                                              
  int& ref = x;  // ref 绑定了 x
                 // ref 和 x 操作的是同一块内存
  ref = 20;      // x 也变成 20

  引用本身不是新对象，只是另一个名字。

  ---
  问题的核心：不是所有东西都有地址

  左值有地址，可以起别名：
  int x = 10;
  int& ref = x;   // OK：x 有地址，能绑

  纯右值没有稳定地址，你怎么给它起别名？
  int& ref = 10;  // 错误：10 在哪？没有地址，绑不住
  int& ref = x+1; // 错误：x+1 的结果在寄存器里，转瞬即逝

  所以引用的规则，本质上是在回答：你要绑的东西，有没有地址？能不能被绑？

  ---
  左值引用 T& ：只绑有稳定地址的东西

  int x = 10;
  int& r = x;    // OK：x 是左值，有地址
  int& r = 10;   // 错：10 是纯右值，没地址

  编译器在检查：你绑的东西是左值吗？

  ---
  右值引用 T&&：专门绑"可以被抢资源"的东西

  右值引用的设计目的不是"给对象起别名"，而是标记这个对象的资源可以被转移。

  int&& r = 10;             // OK：10 是纯右值
  int&& r = std::move(x);   // OK：将亡值，资源可转移
  int&& r = x;              // 错：x 是左值，资源不能随便抢

  编译器在检查：你绑的东西，资源能不能被转移？

  ---
  所以"绑定到不同值类别"的意思是

  ▎ 左值引用和右值引用的区别，不是语法问题，而是语义契约：

  ┌──────────┬────────────────────────────────────────┐
  │ 引用类型 │                契约含义                │
  ├──────────┼────────────────────────────────────────┤
  │ T&       │ 我绑的对象还会继续使用，不能动它的资源 │
  ├──────────┼────────────────────────────────────────┤
  │ T&&      │ 我绑的对象资源可以被转移，你来抢走     │
  └──────────┴────────────────────────────────────────┘

  这个契约决定了编译器在重载决议时选择调用拷贝还是移动：

  void process(Buffer& b)  { /* 拷贝路径 */ }
  void process(Buffer&& b) { /* 移动路径 */ }

  Buffer buf(10);
  process(buf);             // buf 是左值 → 匹配 Buffer&  → 拷贝
  process(std::move(buf));  // 将亡值   → 匹配 Buffer&& → 移动

  ---
  一句话

  ▎ "绑定到不同值类别"= 引用类型是一种声明，声明你绑的对象处于什么状态，从而让编译器决定该复制资源还是转移资源。
```

```text
两个问题分开回答。

引用是别名，但别名可以有多个

int a = 10;
int &r1 = a;   // r1 是 a 的别名
int &r2 = a;   // r2 也是 a 的别名
int &r3 = r1;  // r3 还是 a 的别名

r1 = 20;       // a == 20
r2 = 30;       // a == 30
"别名只能有一个"这个说法是错的，一个对象可以有任意多个别名，就像一个人可以有多个外号。

引用是不是封装后的指针
从语义上看：引用是别名，不是对象，没有自己的地址。

从实现上看：编译器通常确实用指针实现引用。


int a = 10;
int &r = a;
int *p = &a;

// r 和 p 在汇编层面几乎完全一样
// 区别只在语法约束上
但引用在语法层施加了指针没有的限制：

引用	指针
必须初始化	✓	✗
可以为空	✗	✓
初始化后可重新绑定	✗	✓
有自己的地址	✗	✓
使用时需要解引用 *	✗	✓
回到原始问题：T&& 和"别名"有什么关系

Buffer && rr = Buffer(10);
rr 是那个临时对象的别名——它延长了临时对象的生命周期，让一个本来"用完即销毁"的右值，在 rr 的作用域内继续存活。

这是右值引用最重要的作用之一：


{
    Buffer && rr = Buffer(10);   // 临时对象不会立即销毁
    rr.some_method();            // 可以正常使用
}                                // 到这里才销毁
如果没有 rr 绑定它，Buffer(10) 在那行语句结束后就析构了。

一句话总结
引用是别名（语义），底层通常是指针（实现），两者的核心差别在于：指针是独立的对象可以随意操控，引用只是另一个名字受到严格约束。

T&& 只是多了一条约束：只接受右值，其本质仍然是别名。

既然引用时给指定的内存取别名，那么纯右值，没有地址，为什么又可以给它取别名呢？
这是一个很好的矛盾点，答案是：绑定右值时，编译器偷偷做了一件事。

纯右值绑定引用时，编译器自动实体化

const int &r = 10;    // 10 是纯右值，没有地址
编译器实际干的事：


// 你写的
const int &r = 10;

// 编译器做的
int __temp = 10;       // 创建一个临时对象，放到内存里
const int &r = __temp; // r 绑定这个临时对象
纯右值在绑定引用的瞬间被**实体化（materialization）**成了一个真实的内存对象（xvalue），然后引用绑定的是这个实体化后的临时对象，而不是原来那个"没有地址的值"。

这个过程叫做 Temporary Materialization
C++17 正式把它写进标准：

纯右值在需要内存位置的上下文中，会自动转换为 xvalue（将亡值），即拥有地址的临时对象。


纯右值（prvalue）  →  实体化  →  将亡值（xvalue）  →  引用可以绑定
  没有地址                          有地址，但即将销毁
为什么只有 const T& 和 T&& 能绑右值

int &r        = 10;   // 编译错误
const int &r  = 10;   // OK
int &&r       = 10;   // OK
int &r = 10 报错的原因：如果允许，你可以通过 r = 20 修改这个临时对象，但它马上就销毁了，修改毫无意义，编译器直接禁止。

const 和 && 都向编译器表达了"我知道这是临时的"：

const T&：我只读，不会乱改
T&&：我就是要接管这个临时对象的资源
总结
纯右值本身确实没有地址，但绑定引用这个动作会触发实体化，编译器在内存里创建临时对象，引用绑定的是这个临时对象。所以"引用是内存别名"这个说法始终成立，只是右值的内存是编译器替你创建的。
```

```text
右值引用、左值引用、std::move 和完美转发                                                                          
                                                                                                                           
● 左值引用 T&                                                                                                              
                                                                                                                           
  就是传统的引用，绑定左值，保证对象不消亡：                                                                               
                                                                                                                           
  void print(Buffer& b) {                                                                                                  
      // b 肯定活着，可以安全读写                                                                                          
      // 不会转移资源                                                                                                    
  }

  Buffer buf(10);
  print(buf);       // OK
  print(Buffer(10)); // 错：临时对象是右值，绑不上 T&

  主要用途：函数想修改调用方的对象，或避免拷贝的同时不转移资源。

  ---
  右值引用 T&&

  绑定右值（纯右值或将亡值），语义是：我知道这个对象可以被掏空。

  void process(Buffer&& b) {
      // 编译器告诉你：b 的资源可以抢
      // 但注意：b 本身在函数体内是左值！（有名字）
  }

  process(Buffer(10));       // OK：临时对象是纯右值
  process(std::move(buf));   // OK：将亡值
  process(buf);              // 错：buf 是左值

  右值引用最核心的用途就是实现移动构造和移动赋值，上一节已经讲过。

  ---
  std::move

  std::move 不移动任何东西，它只做一件事：

  ▎ 把左值强制转换成将亡值（右值引用），让编译器选择移动路径。

  源码本质就是一个 static_cast：

  // 标准库实现（简化版）
  template<typename T>
  std::remove_reference_t<T>&& move(T&& t) noexcept {
      return static_cast<std::remove_reference_t<T>&&>(t);
  }

  实际效果：

  Buffer buf(10);

  Buffer b1 = buf;            // buf 是左值 → 拷贝构造，buf 还活着
  Buffer b2 = std::move(buf); // 强转成将亡值 → 移动构造，buf 被掏空
  // 此后 buf.data_ == nullptr，不能再用 buf

  使用 std::move 后，原对象处于"有效但未定义"状态，不要再使用它。

  ---
  完美转发 std::forward

  这是最难理解的部分，先说它解决什么问题。

  问题：引用传递中值类别丢失

  void worker(Buffer& b)  { std::cout << "左值路径\n"; }
  void worker(Buffer&& b) { std::cout << "右值路径\n"; }

  // 想写一个包装函数，透明地转发给 worker
  void wrapper(Buffer&& b) {
      worker(b);  // ← 永远走左值路径！因为 b 有名字，是左值
  }

  Buffer buf(10);
  wrapper(std::move(buf)); // 期望走右值路径，实际走了左值路径

  右值引用参数传进函数体后，变成了左值，值类别信息丢失了。

  ---
  解决：std::forward 保留值类别

  template<typename T>
  void wrapper(T&& b) {           // 注意：这里是模板 + T&&，叫"万能引用"
      worker(std::forward<T>(b)); // 转发时恢复原来的值类别
  }

  Buffer buf(10);
  wrapper(buf);             // T=Buffer&  → forward 保持左值 → 走左值路径
  wrapper(std::move(buf));  // T=Buffer&& → forward 保持将亡值 → 走右值路径

  std::forward 的本质：如果原来是左值就保持左值，原来是右值就恢复成右值。

  ---
  万能引用 T&& vs 右值引用 T&&

  写法一样，含义不同：

  void foo(Buffer&& b);   // 普通右值引用：只接受右值

  template<typename T>
  void bar(T&& b);        // 万能引用：左值右值都接受

  区别在于有没有模板类型推导。有推导的 T&& 才是万能引用，它利用了引用折叠规则：

  T = Buffer&   → T&& = Buffer& &&  → 折叠成 Buffer&   （左值引用）
  T = Buffer&&  → T&& = Buffer&& && → 折叠成 Buffer&&  （右值引用）

  ---
  std::move vs std::forward

  ┌──────────────┬────────────────┬──────────────────────┐
  │              │   std::move    │   std::forward<T>    │
  ├──────────────┼────────────────┼──────────────────────┤
  │ 作用         │ 无条件转成右值 │ 有条件保留原值类别   │
  ├──────────────┼────────────────┼──────────────────────┤
  │ 使用场景     │ 明确要转移资源 │ 模板函数透明转发参数 │
  ├──────────────┼────────────────┼──────────────────────┤
  │ 需要模板参数 │ 不需要         │ 需要 <T>             │
  └──────────────┴────────────────┴──────────────────────┘

  // move：我确定要转移，不管来的是什么
  Buffer b2 = std::move(buf);

  // forward：我不知道来的是左值还是右值，原样转发
  template<typename T>
  void wrapper(T&& arg) {
      worker(std::forward<T>(arg));
  }

  ---
  全局串联

  调用方传入左值 buf
      ↓
  wrapper(buf)
      ↓ T 推导为 Buffer&
  void wrapper(Buffer& && arg)  →  折叠成  void wrapper(Buffer& arg)
      ↓ arg 是左值
  std::forward<Buffer&>(arg)  →  保持左值  →  worker(Buffer& b)  拷贝路径


  调用方传入将亡值 std::move(buf)
      ↓
  wrapper(std::move(buf))
      ↓ T 推导为 Buffer&&
  void wrapper(Buffer&& arg)
      ↓ arg 在函数体内是左值，但 T 记录了原来是右值
  std::forward<Buffer&&>(arg)  →  转成右值  →  worker(Buffer&& b)  移动路径

  ---
  一句话总结

  ┌──────────────────────┬──────────────────────────────────┐
  │         概念         │              一句话              │
  ├──────────────────────┼──────────────────────────────────┤
  │ 左值引用 T&          │ 绑左值，不转移资源               │
  ├──────────────────────┼──────────────────────────────────┤
  │ 右值引用 T&&         │ 绑右值，可以转移资源             │
  ├──────────────────────┼──────────────────────────────────┤
  │ std::move            │ 无条件把左值变将亡值，触发移动   │
  ├──────────────────────┼──────────────────────────────────┤
  │ 万能引用 T&&（模板） │ 左值右值都能绑，靠引用折叠实现   │
  ├──────────────────────┼──────────────────────────────────┤
  │ std::forward         │ 在模板中恢复原始值类别，透明转发 │
  └──────────────────────┴──────────────────────────────────┘
```
